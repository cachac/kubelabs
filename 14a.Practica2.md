# 14a. Práctica 2: <!-- omit in toc -->

**Objetivo:** Integrar los conocimientos asociados a la creación de Namespaces, Deployments, Servicios, Almacenamiento Persistente y Networking (Ingress) en un escenario realista.

En esta práctica no se proporcionan los manifiestos YAML ni los comandos exactos para su resolución. Se deben utilizar los laboratorios anteriores como referencia para construir y aplicar los recursos requeridos.

---

## 1. Preparación del Entorno (Namespaces)
Primer objetivo: aislar los recursos de la práctica dentro del cluster.
1. Crear un Namespace con el nombre `practica-app`.
2. Configurar el contexto actual de `kubectl` para que los comandos posteriores se apliquen de forma predeterminada a este nuevo Namespace, evitando la necesidad de especificar el argumento `-n practica-app` en cada ejecución.

> **Nota:** Referencia al Laboratorio **[08. Namespaces](./08.Namespaces.md)** para la creación de un Namespace mediante un archivo YAML y el uso de `kubens` o `kubectl config set-context` para el cambio de contexto.

---

## 2. Almacenamiento (Storage)
Se requiere persistencia de datos para almacenar registros y archivos generados por la aplicación.
1. Crear un **PersistentVolumeClaim (PVC)** con el nombre `app-pvc`.
2. Solicitar una capacidad de almacenamiento de **150Mi**.
3. Especificar el uso de la StorageClass `standard` para el aprovisionamiento dinámico y establecer el modo de acceso `ReadWriteOnce`.

> **Nota:** El aprovisionamiento dinámico genera automáticamente el PersistentVolume (PV). Se recomienda consultar la sección de *Storage Class* en el Laboratorio **[13. Storage](./13.Storage.md)**. Es obligatorio que el manifiesto incluya el atributo `storageClassName: standard`.

---

## 3. Despliegue de la Aplicación (Deployments)
Siguiente despliegue: una aplicación garantizando un esquema de alta disponibilidad.
1. Crear un **Deployment** con el nombre `app-deploy`.
2. Configurar un total de **3 réplicas**.
3. Utilizar la imagen `cachac/kubelabs:3.0` y asignar al contenedor el nombre `frontend-app`.
4. Exponer el contenedor a través del puerto `8080`.
5. Asignar los siguientes límites y solicitudes de recursos:
   - **Requests:** CPU `50m`, Memoria `64Mi`.
   - **Limits:** CPU `150m`, Memoria `128Mi`.
6. Vincular el volumen correspondiente a la reclamación `app-pvc` y montarlo en un directorio del contenedor en la ruta `/usr/share/nginx/html`.

> **Nota:** La estructura de un Deployment, la declaración de puertos y la asignación de recursos se describen en el Laboratorio **[12. Servicios](./12.Servicios.md)**. Aspectos de implementación para `volumes` y `volumeMounts` con un PVC se encuentran documentados en el Laboratorio **[13. Storage](./13.Storage.md)**.

---

## 4. Exposición Interna (Servicios)
Los Pods del Deployment deben agruparse y exponerse internamente en el cluster bajo una dirección estable.
1. Crear un **Service** de tipo `ClusterIP` con el nombre `app-svc`.
2. Establecer el *selector* del servicio para que coincida con las etiquetas (*labels*) de los Pods del Deployment creado anteriormente.
3. Configurar el servicio para que escuche en el puerto `80` a nivel del cluster y dirija el tráfico al `targetPort` correspondiente en los contenedores (`8080`).

> **Nota:** La creación de servicios y su vinculación mediante selectores se detalla en el Laboratorio **[12. Servicios](./12.Servicios.md)**. Omitir el campo `type` en la definición del servicio aplicará el valor predeterminado `ClusterIP`.

---

## 5. Exposición Externa (Networking / Ingress)
Es requisito publicar la aplicación para el acceso externo mediante el uso de un recurso Ingress.
1. Crear un recurso **Ingress** con el nombre `app-ingress`.
2. Definir una regla bajo **Rules** configurando el **Host** para interceptar el dominio `practica.<usuario>.kubelabs.dev` (reemplazar `<usuario>` por el identificador del estudiante).
	2.1 Agrega el subdominio a (dns.kubelabs.dev)[https://dns.kubelabs.dev]
3. Enrutar el tráfico del path raíz (`/`) con pathType `Prefix` hacia el backend definido por el Service `app-svc` en el puerto `80`.

> **Nota:** La declaración de reglas y configuración de backend para los recursos de tipo Ingress se encuentra en el Laboratorio **[14. Networking](./14.Networking.md)**.

---

## 6. Checkpoint de Validación
Tras la aplicación de los manifiestos YAML, se deben ejecutar y corroborar los siguientes puntos:

1. Ejecutar el comando `kubectl get all,pvc,ingress` para visualizar los recursos creados en el Namespace `practica-app`.
2. Validar que existan **3 Pods** en estado **Running**.
3. Confirmar que el PVC `app-pvc` se encuentre en estado **Bound**.
4. Inspeccionar los recursos **Ingress** y **Service** mediante el comando `describe`. Confirmar la consistencia de los puertos y la resolución de *Endpoints* (las IPs de los 3 Pods en ejecución).
5. Como validación final, invocar la aplicación desde un navegador web o mediante el uso de la herramienta `curl` en el host configurado (`practica.<usuario>.kubelabs.dev`), confirmando la respuesta HTTP exitosa.
